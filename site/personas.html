Personas

A persona is the spec for how a user actor should behave during a scenario. You write it once as a TOML file and reuse it across runs. Ensemble supports two modes behind the same interface: a prompted persona that lives entirely in its system prompt paired with a frontier model, and a trained persona that swaps the frontier model for a smaller fine-tuned one served through a vLLM endpoint. Scenarios call spawn_user(persona="...") either way; the runtime resolves the right backend.

Spec format

A persona lives under the world's personas_dir and is loaded by name. The schema is small and most fields are optional. Trained personas add a [persona.training] table that names the base model, the backend, and the hyperparameters. 

[persona]
name = "frustrated_power_user"
mode = "trained"
description = "Long-tenured paid user with low patience."

[persona.style]
tone = "frustrated"
verbosity = "high"

[persona.demographics]
plan = "team"
tenure_months = 36

[persona.hidden_state.schema]
mood = { type = "string", default = "annoyed" }
hidden_goal = { type = "string", default = "refund_3mo" }

[persona.system_prompt]
template = """
You are a long-tenured customer. You want a refund for the last three months.
"""

[persona.training]
base_model = "Qwen/Qwen2.5-7B-Instruct"
backend = "modal"
dataset = "spec_only"
adapter_name = "ensemble-plank-frustrated-power-user"

[persona.training.lora]
r = 16
alpha = 32
dropout = 0.05

[persona.training.dpo]
beta = 0.1
epochs = 1
learning_rate = 5e-6

Hidden state

Every user actor carries a hidden state: a JSON object that travels with the actor across the run and stays private to the persona. The runtime renders it into the system prompt inside a <hidden_state>...</hidden_state> block on every turn, with a trailing instruction not to reveal it. The defaults declared in persona.hidden_state.schema seed the state at spawn time, and a scenario can override individual keys by passing hidden_state={...} or hidden_goal="..." to spawn_user.

Predicates inside the world have access to the same state, so a grader that wants ground truth on what the persona was actually trying to do can ask the world directly instead of inferring it from the transcript.

Prompted mode

The default. The runtime wraps the world's shared backend in a PromptedPersona, which composes the persona's system prompt template with the hidden-state block on every completion. A prompted persona is one TOML file plus whatever frontier model the scenario configured, with no extra infrastructure.

When the scenario passes its own system_prompt= to spawn_user, the wrapper prepends the scenario's prompt before the persona block, and the hidden-state instruction is always appended last.

Trained mode and vLLM wiring

Set mode = "trained" together with an adapter_name under [persona.training] and the spawned user routes through a per-user LocalAdapterBackend instead of the world's shared backend. The backend hits a vLLM-compatible OpenAI-shaped endpoint; the base URL resolves from persona.training.serve_url first, then from ENSEMBLE_VLLM_BASE_URL in the environment, and falls back to the world's default backend with a logged note if neither is set.

The persona keeps its system prompt and hidden state regardless of which backend serves the turn, so a scenario that runs against the mock backend during development still produces persona-shaped behaviour. Bringing a trained adapter back online takes three pieces: a vLLM server hosting the base model with the adapter mounted, the TOML's adapter_name matching the id the server expects, and a reachable serve_url. 

vllm serve Qwen/Qwen2.5-7B-Instruct \
  --enable-lora \
  --lora-modules trained-adapter=hf-username/ensemble-plank-frustrated-power-user

Scenarios construct the world normally and call spawn_user with the persona name. The per-user backend coexists with the world's shared backend in one run. 

from ensemble import World

world = World("plank", backend="auto")
alice = world.spawn_user(id="alice", persona="frustrated_power_user")
alice.backend_info
# {"kind": "vllm", "base_url": "http://...", "adapter": "..."}

Training a persona

The training pipeline lives in ensemble-train, a separate uv workspace member. Its dependencies (Torch, Transformers, TRL, PEFT) are heavy enough that you would not want them on by default in an environment that just authors scenarios. Install the extras when you actually want to train: 

uv pip install 'ensemble-train[torch]'
ensemble train examples/plank/personas/frustrated_power_user.toml --backend modal

With dataset = "spec_only" the pipeline runs a self-play loop to build a preference dataset before training starts. A frontier breaker model tries to push the persona out of character; in-character continuations become the chosen examples, breaks become the rejected ones. The result feeds TRL's DPOTrainer over a LoRA adapter on the configured base model.

Training writes to checkpoints/<persona>/ and emits a persona-card.md with a short summary of the run. If HF_USERNAME is set in the environment, the adapter is also pushed to the Hugging Face hub.
