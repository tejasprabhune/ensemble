Sweeps

A sweep is the same scenario run across many configurations: backends, models, seeds, personas, anything you can express as a CLI flag or an environment variable. This guide walks from the smallest useful sweep up to the resume semantics and the observability subcommands that read the per-cell index.

Why a sweep runner

The question "how does this scenario behave across three seeds and two backends" used to mean a shell loop around ensemble run, a hand-rolled directory naming scheme, and a separate script to collect the results. The sweep runner replaces all of that: one TOML file declares the matrix, one command runs it, and the per-cell output lands in a layout the cross-run subcommands already understand.

Reach for a sweep when you want to declare a matrix of runs, run it with one command, and resume after an interruption without re-paying for the cells that already finished.

The minimal TOML

A sweep is a single TOML file. The smallest useful shape is one scenario plus one axis: 

[sweep]
scenario = "my_world.refund"

[sweep.flags]
backend = ["mock"]

This expands to exactly one cell: the refund scenario, run once against the mock backend. The axis is still useful here because the cell id encodes the backend name, which keeps the on-disk layout consistent once you add more axes.

The full schema

The full sweep file has a [sweep] table for the scenario and run-wide knobs, plus two axis tables. Each axis table contributes one or more dimensions to the cartesian product: 

[sweep]
scenario = "my_world.refund"
world = "my_world"                  # optional; falls through to the scenario's declaration
max_parallel = 4                    # default 1
traces_dir = "traces/refund_sweep"  # default: traces/<scenario_slug>_sweep
package_dir = "."                   # optional; path containing the scenarios package

[sweep.flags]
# Each key becomes a --<key> CLI flag passed to ensemble run.
# Each value list contributes one axis to the cartesian product.
backend = ["mock", "anthropic"]

[sweep.env]
# Each key is exported as an env var for the scenario subprocess.
# Use this when the dimension you want to sweep is not a CLI flag.
AGORA_SEED = ["1", "2", "3"]
MY_TEMPERATURE = ["0.0", "0.7"]

The two axis tables are independent. [sweep.flags] values become CLI flags on the cell's subprocess; [sweep.env] values become environment variables for that subprocess. The runner takes the cartesian product across both tables together, so the file above expands to 2 backends times 3 seeds times 2 temperatures = 12 cells.

If both tables are empty the sweep contains exactly one cell with no overrides. That is useful when you want the on-disk layout and the resume semantics without varying any configuration.

Running

One command runs the whole matrix: 

ensemble sweep run sweep.toml

The runner expands the cartesian product, spawns subprocesses up to max_parallel concurrently, and prints one line per cell to stderr as cells complete: 

[ok] backend-mock__AGORA_SEED-1  scores: refund_issued=1.0
[fail] backend-anthropic__AGORA_SEED-2  scores: <none>
[skipped] backend-mock__AGORA_SEED-2  scores: refund_issued=1.0

After every cell finishes, the runner prints one JSON object on stdout summarising the sweep: 

{"scenario":"my_world.refund","cells":6,"failed":0,"traces_dir":"traces/refund_sweep","index":"traces/refund_sweep/index.jsonl"}

The exit code is 0 when every cell succeeded and 1 when any cell failed. max_parallel = 1 (the default) serialises cells; higher values run them concurrently. Pick the limit based on your backend's rate budget and on whether your scenario uses external resources that do not tolerate concurrent callers.

Cell ids and the on-disk layout

Each cell's axis values combine into a stable, filesystem-safe id that becomes the cell's directory name. With the sweep above and the axes backend = ["mock", "anthropic"] by AGORA_SEED = ["1", "2"], the runner produces: 

traces/refund_sweep/
  index.jsonl
  backend-anthropic__AGORA_SEED-1/
    trace.jsonl
    meta.json
  backend-anthropic__AGORA_SEED-2/
    trace.jsonl
    meta.json
  backend-mock__AGORA_SEED-1/
    trace.jsonl
    meta.json
  backend-mock__AGORA_SEED-2/
    trace.jsonl
    meta.json

index.jsonl is the flat per-cell summary the observability subcommands read: one JSON line per cell with the scenario, the axis values, the scores, the costs, the exit code, and the trace path. meta.json inside each cell directory carries the same data plus an optional error field with the last 2000 bytes of stderr when the subprocess failed, which is the first place to look when a cell shows up as [fail].

Resume semantics

Resume is the default. A cell whose meta.json already exists is skipped on the next run, so a sweep interrupted halfway through can be re-invoked with the same command and only the missing cells run. That is what the [skipped] lines on stderr report.

When you want every cell to re-run, for example after editing the scenario or the world, pass --no-resume: 

ensemble sweep run sweep.toml --no-resume

The flag wipes the resume check, not the per-cell directories themselves; the runner overwrites the trace and meta files as each cell finishes. If you want a clean slate, delete the traces directory before re-running.

Reading sweep results

The sweep's index.jsonl is shaped to be read by the cross-run subcommands. Point them at the sweep directory: 

ensemble runs list --traces-dir traces/refund_sweep
ensemble runs export --traces-dir traces/refund_sweep --format csv > results.csv

ensemble runs list prints a table of cells with scores and costs; ensemble runs export emits the same data as CSV or JSON for downstream plotting. ensemble runs compare takes two cell ids (prefix matching, so you only type the distinguishing part) and diffs their scores side by side. ensemble trace compare opens the two cells' traces in the browser viewer, scroll-synced by tick, when you want to see why the scores differ.

When the cartesian product is wrong

Sometimes two axes should move together rather than independently. A run with model = "claude-haiku-4-5" and temperature = "0.0" may make sense, but pairing it with temperature = "1.0" does not. The sweep schema takes the full cartesian product and has no zip operator.

The workaround is one axis whose values name presets the scenario reads internally: 

[sweep.flags]
preset = ["fast", "balanced", "thorough"]

The scenario reads preset from the CLI flag (or env var) and picks the matching settings itself. This trades schema simplicity for the loss of axis-level reporting in index.jsonl: the cell id is preset-fast rather than carrying the model and temperature as separate columns. An explicit zip operator may land later if the audit's next round finds it load-bearing.
