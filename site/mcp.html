MCP

Ensemble speaks the Model Context Protocol two ways. It can expose a world's tool catalogue to any MCP-aware client so an external agent can call those tools against a controlled, typed world. It can also hand a running scenario over to an external client by replacing one named agent slot with a proxy, while the scheduler keeps routing messages and writing the trace. Both modes share the same server; the flag you pass to ensemble mcp serve decides which one you get.

Tools-only mode

The simplest setup. Point ensemble mcp serve at a world and the server publishes every tool in that world's registry over MCP. The client lists tools, calls them, and gets back the same JSON the runtime would hand a local agent. No scenario runs in the background, no other actors exist; this is the mode to reach for when you want a real production agent (Claude Code, Claude Desktop, Codex, a custom MCP host) to exercise the world's tool surface directly. 

ensemble mcp serve --world agora

Per-tool timeouts and resource locks still apply, since dispatch goes through the same ResourceManager the agent loop uses.

Scenario-driving mode

Pass --scenario and --as-agent and the server boots the scenario in a background thread, then substitutes one named agent slot with an MCP proxy. The scheduler keeps routing messages to that slot's inbox; the responses come from the connected client instead of the LLM the scenario configured. The trace records the slot's activity under its actor id, so a grader written against a normal rollout still works against an MCP-driven one. 

ensemble worlds add agora examples/agora
ensemble mcp serve --world agora --scenario agora.refund_storm --as-agent rep1

With --backend mock (the default) the other actors use scripted responses; pass --backend anthropic to give the non-MCP actors a real model. When the until predicate fires the scheduler halts and the server exits.

The meta tools

In scenario-driving mode the client sees three kinds of tools:

  • The world's tools, filtered to the slot's allowed list.
  • inbox_recv, which returns the next message routed to the slot, or {empty: true} when the inbox is drained.
  • agent_say, which sends a message from the slot to another actor by id.

The two meta tools are what make the slot driveable from a generic MCP client. The client loop is: call inbox_recv, decide what to do, dispatch zero or more world tools, then either agent_say a reply or loop back to inbox_recv. 

# client.py, driving a slot through ensemble's MCP server.
import asyncio
from mcp import ClientSession
from mcp.client.stdio import StdioServerParameters, stdio_client

params = StdioServerParameters(
    command="ensemble",
    args=["mcp", "serve",
          "--world", "agora",
          "--scenario", "agora.refund_storm",
          "--as-agent", "rep1"],
)

async def main():
    async with stdio_client(params) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()
            tools = await session.list_tools()
            print([t.name for t in tools.tools])
            msg = await session.call_tool("inbox_recv", arguments={})
            print(msg.content[0].text)

asyncio.run(main())

Transport and implementation notes

  • Only stdio is supported today. SSE and streamable HTTP land in a follow-up; the protocol shape is identical.
  • The server uses the official Python MCP SDK (mcp.server.lowlevel.Server). The rust rmcp crate could host the server itself, but plugin tools are already Python callables and an rmcp host would re-acquire the GIL on every dispatch. The inline note at python/ensemble/mcp_server.py tracks the upgrade path.
  • Per-tool timeouts and resource locks apply to MCP-dispatched calls in both modes.
