Training runs

A training run is a persona fine-tuning job tracked by Stage. This page covers how to start one, what Stage records during training, and how to read the training detail page to evaluate whether the run succeeded.

Starting a training run

Training is driven by the ensemble train command, which takes a persona TOML path and a compute backend: 

ensemble train examples/agora/personas/frustrated_power_user.toml --backend modal

When ENSEMBLE_STAGE_API_KEY and ENSEMBLE_STAGE_PROJECT are set, the training pipeline creates a training run entity on Stage before the job starts and prints the URL: 

Training run: https://ensemble-stage.fly.dev/myorg/myproject/training_runs/019b2c3d-...

The pipeline streams step-level metrics to Stage as training progresses. Each flush is a batch of metric points sent to POST /v1/training_runs/{id}/metrics. Flushes happen on a 30-second background timer so individual training steps are not gated on network I/O.

When training completes (or fails), the pipeline posts a final status update with the final_metrics object and, if an adapter was uploaded, the artifact_uri.

The persona TOML training block

Training configuration lives in the persona TOML under [persona.training]. The full schema: 

[persona.training]
base_model = "Qwen/Qwen2.5-7B-Instruct"  # required
backend = "modal"                          # modal | skypilot | local
dataset = "spec_only"                      # dataset generation strategy
hf_namespace_env = "HF_USERNAME"           # env var holding the HF username
adapter_name = "ensemble-agora-frustrated-power-user"  # HuggingFace adapter id
serve_url = "http://127.0.0.1:8000/v1"    # vLLM endpoint for inference after training

[persona.training.lora]
r = 16                  # LoRA rank
alpha = 32              # LoRA alpha (scaling factor = alpha/r)
dropout = 0.05
target_modules = ["q_proj", "v_proj"]

[persona.training.dpo]
beta = 0.1              # KL penalty strength; higher values stay closer to reference
epochs = 1
learning_rate = 5e-6
batch_size = 4
max_length = 2048       # maximum sequence length in tokens

[persona.training.self_play]
breaker_model = "claude-sonnet-4-5"  # the adversarial model that generates hard cases
rollouts = 200           # number of self-play trajectories to generate
break_judge = "claude-sonnet-4-5"    # model used to score trajectory outcomes

The dataset field controls how the preference dataset is generated. "spec_only" derives pairs from the persona specification without running the full scenario. Other strategies (such as "self_play") run the [persona.training.self_play] block to produce trajectories, then use the judge model to score chosen vs. rejected.

The adapter_name becomes the HuggingFace repository name under the namespace from hf_namespace_env. After training completes, the full HuggingFace URI is set as the training run's artifact_uri on Stage.

Compute backends

modal (default)

Submits the training job to Modal cloud. Requires the modal Python package (uv pip install modal) and a Modal account configured via modal token new. GPU allocation and the training image are defined in the training package; the persona TOML controls only the hyperparameters.

When modal is not installed, the pipeline writes a DRY_RUN.json next to the checkpoint directory showing what would have been submitted. Use this to validate the config without spending credits.

skypilot

Writes a SkyPilot YAML and either launches it via sky launch (if sky is on PATH) or prints the command for you to run manually. Use for multi-cloud training or when you need control over the cluster configuration.

local

Runs the training loop in-process on the current machine. Requires a GPU and the training dependencies installed (uv pip install 'ensemble-train[torch]'). Writes the adapter under checkpoints/<persona_name>/. Suitable for small models or for debugging the training code.

Stage API endpoints for training runs

The following API calls are relevant to training runs. All require a push-scoped API key unless noted.

MethodPathDescription
POST /v1/projects/{org}/{proj}/training_runs Create a training run. Body: persona_name, base_model, hyperparameters.
GET /v1/training_runs/{id} Get run metadata including status, hyperparameters, final_metrics, artifact_uri.
POST /v1/training_runs/{id}/metrics Append step metrics. Body: metrics array of {step, metric_name, value}. Up to 1000 per request.
GET /v1/training_runs/{id}/metrics Get all metric points. Poll this endpoint to follow a live run.
POST /v1/training_runs/{id}/status Update status. Body: status (running|completed|failed|cancelled), optional final_metrics and artifact_uri.
GET /v1/projects/{org}/{proj}/training_runs List training runs, newest first.

Duplicate (step, metric_name) pairs within the same training run are silently ignored on append.

Reading the training detail page on Stage

The training detail page at /org/project/training_runs/<id> is designed to answer five questions after a run completes: did the loss go down, did eval improve, is this run better than past runs for this persona, did the adapter actually improve eval outcomes versus the prompted baseline, and where is the artifact.

Header

The header shows the full training run UUID, status badge, persona name, base model, and timestamps. When training completes with an artifact, the artifact URI appears alongside a copy button. The URI is the permanent HuggingFace or Modal volume address for the adapter.

Loss curves

The first thing to check is whether the loss curves fell. The page renders one SVG chart per metric name; train_loss and eval_loss share a chart when both are present. The x-axis is the training step; the y-axis is the metric value.

The y-axis has a log-scale toggle. Linear scale shows absolute changes; log scale shows proportional changes and makes early large drops and late small refinements comparable in height. The toggle state is preserved in the URL hash so you can share a specific view.

Three annotations overlay the chart automatically:

  • A vertical line at the step with the lowest eval loss, labeled "Best eval @ step N." This is the checkpoint most likely to generalize. If this line is early in the run, the model may have overfit after that point.
  • A marker at the last recorded eval point labeled with its value. If the final eval is significantly worse than the best eval, the model overfit in later steps.
  • A horizontal band labeled "Plateau" over any region where the loss stayed within 1% for 20% or more of total steps. A plateau indicates that further training on this configuration will not improve the result; consider adjusting the learning rate or switching to a different checkpoint.

While training is in progress, the chart polls every five seconds and appends new points without redrawing from scratch.

Hyperparameters and final metrics

Two side-by-side tables sit below the loss curves. The left table lists hyperparameters: base model, learning rate, batch size, epochs, LoRA rank (r), LoRA alpha, DPO beta, and any other fields the training backend stored. The right table lists final metrics: final train loss, final eval loss, best eval loss and the step it occurred at, total steps, wall time, and total cost.

Reading both tables together tells you what was trained and how well it converged. Compare the best eval loss to the final eval loss; a large gap suggests the training continued past the optimal point.

Comparison to past training runs

The comparison chart overlays the current run's eval_loss curve with the three or five most recent prior training runs for the same persona name. The current run is drawn opaque in the accent color; older runs are drawn at half opacity in a muted color. A legend below the chart labels each line with its training run ID (first eight characters), date, and the key hyperparameter difference from the current run, for example "lr=5e-6 vs current 1e-5."

Use this view to determine the effect of your hyperparameter change. If the current run's eval loss converges lower or faster than prior runs, the change helped. If the lines are nearly identical, the hyperparameter does not matter much for this persona. If the current run is worse, something regressed.

If this is the first training run for this persona, the section shows: "This is the first training run for this persona. The comparison section will populate once additional runs exist."

Baseline comparison

This is the most important section on the page. It shows whether the trained adapter actually improved eval outcomes compared to the prompted baseline. Stage constructs this comparison by finding eval runs in the same project that used the adapter from this training run, then finding matching eval runs that used the prompted version of the same persona on the same scenario and model.

The result is a table sorted by absolute delta descending, with one row per (scenario, model) pair:

Column What it shows
Scenario The scenario the eval runs executed.
Model The model used for the agent (not the persona).
Baseline Mean primary outcome score for prompted-persona eval runs.
Trained Mean primary outcome score for trained-adapter eval runs.
Delta Trained minus baseline. Green for positive, red for negative, intensity by magnitude.
p-value Statistical significance when enough samples exist. Absent if insufficient data.
Compare Link to the Stage comparison view for a representative run pair.

A positive delta means the trained persona performed better on that scenario and model combination than the prompted baseline. A negative delta is a regression; investigate the specific run pairs using the compare link.

The baseline comparison requires two things to work. First, eval runs must record which adapter they used (the adapter_uri metadata field). Second, matching prompted-persona eval runs must exist in the same project. If the linkage is not yet wired, the section says so and links to the personas reference for how to close the loop.

Artifact details

A section at the bottom shows the artifact URI, a copy button, and the artifact size. A code snippet shows how to add the adapter to the persona TOML and run an eval scenario against it: 

# In the persona TOML
[persona.training]
adapter_name = "your-hf-username/ensemble-agora-frustrated-power-user"
serve_url = "http://your-vllm-host:8000/v1"
ensemble run agora.refund_storm --backend auto

If training is still in progress, the section shows "Training in progress." If training failed without producing an artifact, it shows "Training failed; no artifact produced."

Preference dataset preview

When the training run's metadata includes a reference to the preference dataset used for DPO, a collapsible section shows the pair count and three example pairs with chosen and rejected responses side by side. This gives you a feel for what signal the model trained on without having to fetch the dataset separately. If the dataset reference was not recorded, this section is hidden.

Closing the eval loop

The baseline comparison on the training detail page is the payoff of the post-training loop, but it requires all three steps to be wired together correctly.

  1. Train. ensemble train persona.toml with Stage configured creates a training run and uploads the adapter to HuggingFace. artifact_uri is set on the training run entity when the upload completes.
  2. Configure the persona to use the adapter. Set persona.training.serve_url to your vLLM endpoint and adapter_name to the HuggingFace adapter id. Restart the vLLM server with --lora-modules adapter_name=hf-user/adapter-repo.
  3. Run eval scenarios with the trained persona. ensemble run agora.refund_storm routes user actors with mode = "trained" through the adapter automatically. The eval runs land on Stage in the same project.
  4. Run the same scenarios with the prompted persona. Temporarily set mode = "prompted" in the persona TOML (or use a copy of the file with mode changed) and run the same scenarios. These form the baseline.
  5. Link the eval runs to the adapter. Stage finds trained eval runs by matching the run's adapter_uri metadata field to the training run's artifact_uri. Ensure the ensemble integration records this field when creating the run.

When all five steps are complete, the baseline comparison table on the training detail page populates automatically.
