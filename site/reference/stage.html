Stage

Stage is ensemble's cloud observability backend. It stores run histories, sweep results, and training metrics so you can share them with teammates, compare runs across machines, and revisit results long after the local files have been cleaned up. Stage is entirely optional: ensemble always writes a complete local JSONL trace first, and Stage receives the same events in parallel. If Stage is unreachable the local run is unaffected.

When to use Stage

The local trace file and the ensemble runs subcommands are the right tools when you are iterating alone on a single machine. You can open any run in the viewer, diff two runs with ensemble runs compare, and export the index for downstream scripts without touching Stage at all.

Stage adds value when any of the following apply. You want to share a run with a colleague who does not have the file on their machine. You run scenarios on more than one machine and want a unified history without manually syncing files. You are running sweeps and want a persistent matrix view that outlives the working directory. You are training a persona and want metric charts across training steps. You are writing a script that needs to query run results without reading JSONL files from disk.

Stage and local traces are complementary. Every event goes to the local JSONL file first; Stage receives them asynchronously in batches. If you have Stage configured and a run fails to reach it, the local trace is still complete and you can push it retroactively.

Setting up

Stage uses GitHub OAuth for identity. Open ensemble-stage.fly.dev and click "Sign in with GitHub." After you authorize the app, Stage creates your account and uses your GitHub login as your personal org slug. You land on your org page at /your-github-login.

Create a project from your org page. The inline form at the top of the page takes a slug (lowercase letters, digits, and hyphens), a display name, and an optional description. Click "Create project." Stage redirects you to the new project page at /your-login/your-project-slug.

Create an API key by going to /me. Give the key a name you will recognize ("laptop" or "ci" work well), leave the scope as "push," and click "Create key." The full key value appears exactly once in a banner at the top of the page. Copy it immediately and store it somewhere safe, such as your shell profile or a password manager. The server stores only a SHA-256 hash, so the value cannot be recovered after you leave the page.

To revoke a key, return to /me and click "Revoke" next to the key's name.

Configuring ensemble

The two environment variables ENSEMBLE_STAGE_API_KEY and ENSEMBLE_STAGE_PROJECT are all you need for the common case. Set them in your shell profile or export them in the session before running scenarios: 

export ENSEMBLE_STAGE_API_KEY=stage_sk_...
export ENSEMBLE_STAGE_PROJECT=myorg/myproject

For a repository where everyone should target the same Stage project, write a .stage.toml file at the repository root and commit it. Each developer still provides their own ENSEMBLE_STAGE_API_KEY independently; the file only records which project to target. 

[stage]
project = "myorg/myproject"

Configuration resolves through three levels of precedence. An explicit Stage(...) object passed to World() wins. Environment variables win over config files. Within config files, the API key comes from ~/.ensemble/stage.toml under [credentials], and the project slug comes from .stage.toml in the working directory under [stage].

To configure Stage in Python rather than through environment variables, pass a Stage object directly to World(): 

from ensemble.stage import Stage
from ensemble import World

stage = Stage(project="myorg/myproject", api_key="stage_sk_...")
world = World("agora", stage=stage)

To disable Stage for a single command without unsetting environment variables, set ENSEMBLE_STAGE_ENABLED=0: 

ENSEMBLE_STAGE_ENABLED=0 ensemble run agora.refund_storm

Running a scenario

Once the environment variables are set, running a scenario works exactly as before. Stage integration is transparent: ensemble creates the run entity on Stage before the scenario starts, streams events in the background, and prints the Stage URL alongside the run id on stderr. 

ensemble run agora.refund_storm
# Run id: 019e4ac0-9075-7113-a038-dfeac521846f
# Stage:  https://ensemble-stage.fly.dev/myorg/myproject/runs/019e4ac0-...

The Stage URL is the permanent address for that run. You can open it while the run is still in progress to watch events arrive, or share it with a colleague. The run stays accessible on Stage after the local trace file is cleaned up.

Events reach Stage asynchronously in batches of up to 100. When the scenario finishes, any remaining buffered events are flushed with a five-second timeout before the run is marked completed. If the flush fails, ensemble prints a warning with an event count and an ensemble stage push command you can run retroactively.

The run detail page

Opening a run URL on Stage shows the full trace in a structured layout. The page is divided into several sections that correspond to different aspects of what happened during the scenario.

The timeline at the top is a scrubber over the trace. Each tick represents one event. Clicking a tick or dragging the slider updates all panels below to show the state at that point in the run. The tick count and the scenario name appear in the header.

The actors panel shows which actors (users and agents) sent messages and when. Each message appears with its sender, recipient, and text, annotated with the tick it occurred at. This panel gives a narrative view of the conversation as it unfolded.

The tool calls panel lists every tool dispatch in order: the tool name, the arguments the agent passed, and the result the world returned. For tools that emit diffs, a collapsed diff indicator appears next to the result.

The state changes panel shows the world state diffs that tools produced. Each diff entry identifies the table, row, and field that changed, plus the old and new values. This panel confirms that a refund landed in the database, or that a ticket status moved from "open" to "escalated."

The hidden state section shows each user actor's hidden goal and the values of their hidden state fields at each tick. This is the private context the grader evaluates after the run; seeing it alongside the conversation lets you audit whether the agent resolved the user's unstated objective.

The predicates section shows the final result of every grader predicate evaluation: which predicates fired, which did not, and the score each one contributed. If a run scored unexpectedly, this section shows exactly which predicate caused the divergence.

The outcome section summarizes the final scores as a key-value table, matching the JSON object the scenario yielded. The cost section shows the total token and USD costs broken down by actor and by call.

Running a sweep

Sweeps on Stage work through the same ensemble sweep run command you use locally. When Stage is configured, ensemble creates a parent sweep entity before launching cells and associates each cell run with the parent. The sweep URL is printed at startup. 

ensemble sweep run sweep.toml
# Sweep id: 019a1b2c-...
# Stage:    https://ensemble-stage.fly.dev/myorg/myproject/sweeps/019a1b2c-...

The sweep detail page has several sections. A KPI strip at the top shows run counts, completion percentage, total cost, and mean outcome with standard error. It polls every two seconds while the sweep is running. Below the KPI strip is a matrix view with axis dropdowns (for choosing which dimensions appear as rows and columns), a variance toggle, and cells that each show a status indicator, mean score, and a box plot glyph. A cost-over-time line chart shows cumulative USD spend as runs complete. A per-axis breakdown table covers axes not shown in the matrix. At the bottom, a flat runs list is filterable by matrix cell or by axis value.

The action row at the top of the page lets you share the sweep URL, open a comparison picker for two runs in the sweep, or cancel the sweep if it is still running.

See Reading the sweep detail page on Stage for a complete description of how to interpret each section.

Training

When you run ensemble train with Stage configured, ensemble creates a training run entity and streams step metrics as training progresses. The training run URL is printed at startup. 

ensemble train examples/agora/personas/frustrated_power_user.toml --backend modal
# Training run: https://ensemble-stage.fly.dev/myorg/myproject/training_runs/019b2c3d-...

The training detail page shows loss curves for each metric (train_loss and eval_loss share a chart; others are separate), with annotations for the best eval checkpoint and any training plateau. A log-scale toggle switches the y-axis. The page polls every five seconds while training is running.

Below the charts are two side-by-side tables: hyperparameters (learning rate, batch size, LoRA rank, DPO beta, etc.) and final metrics (best eval loss, total steps, wall time, cost). A comparison section overlays the current run's loss curve with the most recent prior training runs for the same persona. A baseline comparison table shows whether the trained adapter improved eval outcomes relative to the prompted persona on the same scenarios and models.

Metric flushing uses a background thread with a 30-second timer so individual training steps are not blocked by network I/O. If the training process exits uncleanly, buffered metrics are flushed at process teardown.

See Reading the training detail page on Stage in the personas reference for a full description of the baseline comparison mechanism.

Comparing runs

Stage lets you compare any two runs side by side. From the runs list page for a project, select two runs by clicking their checkboxes, then press c. Stage opens a comparison view with both runs' timelines synchronized by tick.

The comparison view shows the two traces in parallel columns. Scrubbing the shared timeline updates both columns simultaneously, so you can see how the agents diverged at the same point in the run. The scores table at the bottom shows both runs' grader outputs side by side with differences highlighted.

You can navigate directly to a comparison by providing two run ids in the query string: /myorg/myproject/compare?a=019e4ac0&b=019f5bd1. This URL is shareable: a colleague with access to the project sees the same view without needing to reproduce either run locally.

Deleting runs

To delete runs from the list page, check one or more rows and press d (or click the delete button that appears in the filter bar). Stage asks for confirmation before sending the delete requests, then removes the rows from the table. Deletions are permanent.

To delete a single run from its detail page, click the "delete" button in the run header. After confirmation Stage navigates back to the project runs list.

Sharing

Any run, sweep, or training run detail page has a "Share" button in the top-right corner. Clicking it generates a share token and returns a URL of the form /share/<token>. Anyone who has that URL can view the run without a Stage account or GitHub login. Share tokens give read-only access to the specific run they were generated for and do not grant access to the rest of the project.

Share tokens do not expire by default. To revoke access, open the run detail page, click "Manage shares," and delete the token. The URL stops working immediately.

Bulk push

Local traces that accumulated before Stage was configured can be pushed retroactively. The push command reads each run directory's meta.json, checks whether that run id already exists on Stage, and skips it if so. Running push twice on the same directory is safe. 

# Push every trace under traces/:
ensemble stage push traces/

# Push one specific run directory:
ensemble stage push traces/019e4ac0-9075-7113-a038-dfeac521846f/

# Push using a glob pattern:
ensemble stage push "traces/*/trace.jsonl"

Push prints a summary when it finishes: N pushed, M skipped, K failed. Failures are printed individually with the run id and error so you can retry them selectively.

Stage API

Stage exposes a REST API at /v1. All endpoints return JSON. Pass a push-scoped API key as a Bearer token: 

curl -s -H "Authorization: Bearer $ENSEMBLE_STAGE_API_KEY" \
  "https://ensemble-stage.fly.dev/v1/projects/myorg/myproject/runs?limit=20"

Troubleshooting

Events failed to reach Stage. If events fail to push during a run, the last line of output includes a count and an ensemble stage push command with the trace path. The run is locally complete. Run that command afterwards to send the events retroactively.

Credential errors. If ensemble stage whoami returns an unauthorized error, the stored key may have been revoked or the wrong key is in the environment. Run ensemble stage logout then ensemble stage login to get a fresh key.

Wrong project. If runs appear in the wrong Stage project, check .stage.toml in the current directory and the ENSEMBLE_STAGE_PROJECT environment variable. An explicit Stage object passed to World() takes priority over both.

Self-hosted Stage. Set ENSEMBLE_STAGE_BASE_URL to your instance's URL. The same variable can go in ~/.ensemble/stage.toml under [credentials].base_url, or in the project's .stage.toml under [stage].base_url.
