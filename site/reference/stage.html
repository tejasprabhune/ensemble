Stage

Stage is ensemble's cloud observability backend. It stores run histories, sweep results, and training metrics so you can share them with teammates, compare runs across machines, and revisit results long after the local files have been cleaned up. It is entirely optional: ensemble always writes a local JSONL trace, and Stage receives the same events in parallel.

When to use Stage

The local trace file is the right choice for iteration on a single machine. You can open it in the viewer, diff two runs with ensemble runs compare, and build your grader against it. You do not need Stage for any of that.

Use Stage when you want any of the following:

  • Share a run URL with a colleague who does not have the trace file on their machine.
  • See a merged history of runs across multiple machines or researchers in a single ensemble runs list view.
  • Track sweep results and training metrics in a persistent store that outlives the working directory.
  • Push older local traces that accumulated before Stage was set up.

Stage and local traces are complementary, not competing. Every event goes to the local JSONL file first; Stage receives them in parallel over an asynchronous buffered connection. If Stage is unreachable, the local trace is unaffected.

Authentication

Stage uses GitHub OAuth for identity. Sign in at ensemble-stage.fly.dev by clicking "Sign in with GitHub." After you authorize, Stage creates your account, using your GitHub login as your personal org slug, and redirects you to your org page.

Once signed in, go to /me and create a push-scoped API key. Give it a name you will recognize (for example "laptop" or "workstation"), leave the scope as "push," and click "Create key." The raw key value appears exactly once in a banner. Copy it and store it somewhere safe; the server stores only a SHA-256 hash and the value cannot be recovered. 

# After copying the key from /me:
export ENSEMBLE_STAGE_API_KEY=stage_sk_...

# Verify it works:
curl -s -H "Authorization: Bearer $ENSEMBLE_STAGE_API_KEY" \
  https://ensemble-stage.fly.dev/v1/me
# {"id": 1, "github_login": "tejasprabhune", "default_org_slug": "tejasprabhune"}

The key lives wherever you store it (shell profile, password manager, CI secret). It is never written to the project's .stage.toml or committed to version control.

To revoke access, return to /me and click "Revoke" next to the key.

Configuration

Configuration follows a three-level precedence:

  1. An explicit Stage(...) object passed to World().
  2. Environment variables: ENSEMBLE_STAGE_API_KEY, ENSEMBLE_STAGE_PROJECT, ENSEMBLE_STAGE_BASE_URL (optional), ENSEMBLE_STAGE_ENABLED (set to 0 to disable).
  3. Config files: the API key comes from ~/.ensemble/stage.toml under [credentials]; the project slug comes from .stage.toml in the working directory under [stage].

The common case is to set two environment variables and let the rest happen automatically: 

export ENSEMBLE_STAGE_API_KEY=stage_sk_...
export ENSEMBLE_STAGE_PROJECT=myorg/popcornbench
ensemble run plank.refund_storm

For a project where everyone should target the same Stage project, write .stage.toml at the repository root and commit it: 

[stage]
project = "myorg/popcornbench"

Each developer still needs ENSEMBLE_STAGE_API_KEY or a personal credentials file; the project file only records which Stage project to target.

To disable Stage for one command while it is otherwise configured, set ENSEMBLE_STAGE_ENABLED=0: 

ENSEMBLE_STAGE_ENABLED=0 ensemble run plank.refund_storm

To configure Stage explicitly in Python: 

from ensemble.stage import Stage
from ensemble import World

stage = Stage(project="myorg/popcornbench", api_key="stage_sk_...")
world = World("plank", stage=stage)

Projects

Every run belongs to a Stage project of the form org_slug/project_slug, where org_slug is typically your GitHub login for personal projects.

Create a project through the Stage web UI. After signing in, your personal org page at https://ensemble-stage.fly.dev/your-login shows an inline form. Fill in a slug (lowercase letters, digits, and hyphens), a display name, and click "Create project." Stage redirects you to the new project page.

Once the project exists, set the project slug as an environment variable so ensemble targets it automatically: 

export ENSEMBLE_STAGE_PROJECT=myorg/popcornbench

For a repository where everyone should target the same Stage project, write .stage.toml at the root and commit it: 

[stage]
project = "myorg/popcornbench"

The environment variable takes priority over the config file; each developer still provides their own ENSEMBLE_STAGE_API_KEY.

Runs, sweeps, and training runs

Stage tracks three entity types, each with its own URL and status lifecycle.

Runs. When Stage is configured, ensemble run creates a run entity before the scenario starts and prints its URL to stderr: 

Run id: 019542a3-4e7b-7000-8e1d-3f9a1c2d5e6f
Stage:  https://stage.ensemble.sh/myorg/popcornbench/runs/019542a3-...

Events stream to Stage throughout the run in the background. The local trace file receives every event first; Stage receives them asynchronously in batches of up to 100. When the scenario ends, remaining events are flushed with a five-second timeout and the run is marked completed. If Stage is unreachable, a warning with an event count and a ensemble stage push hint is printed, and the local trace is complete regardless.

Sweeps. ensemble sweep run creates a parent sweep entity on Stage before launching cells. Each child run includes the sweep's id, so Stage can group them. The sweep URL is printed at startup and the final status is posted when all cells finish.

Training runs. ensemble train creates a training run entity and streams step metrics during training. The training run URL is printed at startup. Metric flushing uses a background thread with a 30-second timer so individual training steps are not blocked by network I/O.

Bulk push

Traces that accumulated before Stage was configured can be pushed retroactively. The push command reads each run's meta.json, checks whether that run id is already on Stage, and skips it if so. Idempotent: running push twice on the same directory is safe. 

# Push every trace under traces/:
ensemble stage push traces/

# Push one specific run directory:
ensemble stage push traces/019542a3-4e7b-7000-8e1d-3f9a1c2d5e6f/

# Push using a glob pattern:
ensemble stage push "traces/*/trace.jsonl"

Push reports a count at the end: N pushed, M skipped, K failed.

Troubleshooting

Stage push failures. If events fail to reach Stage during a run, the last line of output includes a count and an ensemble stage push hint with the trace path. The run is still locally complete. Run push afterwards to retroactively send the events.

Credential errors. If ensemble stage whoami returns an unauthorized error, the stored key may have been revoked. Run ensemble stage logout then ensemble stage login to get a fresh key.

Wrong project. If runs are appearing in the wrong Stage project, check .stage.toml in the current directory and the ENSEMBLE_STAGE_PROJECT environment variable. Explicit Stage config in Python beats both.

Self-hosted Stage. Set ENSEMBLE_STAGE_BASE_URL to your instance's URL. The same variable can go in ~/.ensemble/stage.toml under [credentials].base_url or in the project's .stage.toml under [stage].base_url.
