Sweeps

A sweep is the same scenario run across a cartesian product of configurations: backends, models, seeds, personas, or anything else expressible as a CLI flag or an environment variable. The runner produces one trace per cell, captures scores and costs in a per-cell meta.json, and appends a row per cell to a flat index the cross-run observability subcommands read.

Reach for this page when you want to declare a matrix of runs in TOML, run it with one command, and resume after an interruption without re-paying for the completed cells.

The TOML schema

A sweep is a single TOML file. The minimal shape is one scenario plus one axis: 

[sweep]
scenario = "my_world.refund"

[sweep.flags]
backend = ["mock"]

The full schema:

[sweep]
scenario = "my_world.refund"
world = "my_world"               # optional; falls through to the scenario's declaration
max_parallel = 4                 # default 1
traces_dir = "traces/refund_sweep"   # default: traces/<scenario_slug>_sweep
package_dir = "."                # optional; path containing the scenarios package

[sweep.flags]
# Each key becomes a --<key> CLI flag passed to ensemble run.
# Each value list contributes one axis to the cartesian product.
backend = ["mock", "anthropic"]

[sweep.env]
# Each key is exported as an env var for the scenario subprocess.
# Use this when the dimension you want to sweep is not a CLI flag.
AGORA_SEED = ["1", "2", "3"]
MY_TEMPERATURE = ["0.0", "0.7"]

The two axis tables are independent: [sweep.flags] values become CLI flags on the cell, [sweep.env] values become environment variables. The runner takes the cartesian product across both.

If both tables are empty, the sweep contains exactly one cell with no overrides.

Cell identifiers and output layout

Each cell's axis values combine into a stable, filesystem-safe id that becomes the cell's directory name. With the sweep above and the axes backend = ["mock", "anthropic"] x AGORA_SEED = ["1", "2"], the runner produces: 

traces/refund_sweep/
  index.jsonl
  backend-anthropic__AGORA_SEED-1/
    trace.jsonl
    meta.json
  backend-anthropic__AGORA_SEED-2/
    trace.jsonl
    meta.json
  backend-mock__AGORA_SEED-1/
    trace.jsonl
    meta.json
  backend-mock__AGORA_SEED-2/
    trace.jsonl
    meta.json

index.jsonl is the flat per-cell summary the observability subcommands read: one JSON line per cell containing the scenario, the cell's axis values, the scores, the costs, the exit code, and the trace path.

meta.json carries the same data the index row does, plus an optional error field with the last 2000 bytes of the cell's stderr when the subprocess failed.

Running, resuming, and parallelism

ensemble sweep run sweep.toml

The runner expands the cartesian product, spawns subprocesses up to max_parallel concurrently, and prints one line per cell to stderr as cells complete: 

[ok] backend-mock__AGORA_SEED-1  scores: refund_issued=1.0
[fail] backend-anthropic__AGORA_SEED-2  scores: <none>
[skipped] backend-mock__AGORA_SEED-2  scores: refund_issued=1.0

max_parallel = 1 (the default) serialises cells; higher values run cells concurrently. Pick the limit based on your backend's rate budget and on whether your scenario uses external resources that do not tolerate concurrency.

Resume is the default. A cell whose meta.json already exists is skipped, so a sweep interrupted halfway through can be re-run without re-paying for the completed cells. Pass --no-resume to force every cell to re-run.

The summary line on stdout (after all cells complete) is one JSON object with the total cell count, the failed count, the traces directory, and the index path: 

{"scenario":"my_world.refund","cells":6,"failed":0,"traces_dir":"traces/refund_sweep","index":"traces/refund_sweep/index.jsonl"}

The exit code is 0 when every cell succeeded and 1 when any cell failed.

Reading sweep results

The sweep's index.jsonl is a per-cell flat file. The cross-run observability subcommands (see observability.md) can be pointed at the sweep's directory: 

ensemble runs list --traces-dir traces/refund_sweep
ensemble runs export --traces-dir traces/refund_sweep --format csv > results.csv

ensemble runs compare takes two cell ids (or unique prefixes) and diffs their scores side by side. ensemble trace compare opens the two cells' traces in the browser viewer, scroll-synced by tick.

When the cartesian product is wrong

If two axes should move together rather than independently, the sweep schema does not support a zipped product directly. The workaround is one cell per combination encoded as a single axis value: 

[sweep.flags]
preset = ["fast", "balanced", "thorough"]

Then the scenario reads the preset name from the CLI flag or env var and picks the matching settings internally. This trades schema simplicity for the loss of axis-level reporting; an explicit zip operator may land later if the audit's next round finds it load-bearing.

Stage coordination

When Stage is configured, ensemble sweep run creates a parent sweep entity on Stage before launching any cells and prints the sweep URL to stderr: 

Sweep:  https://ensemble-stage.fly.dev/myorg/popcornbench/sweeps/019542b0-...

Each child run is created on Stage with a reference to the parent sweep id, so Stage can group them and track aggregate progress. When the entire sweep finishes (or fails), the sweep entity is marked with its terminal status.

If Stage is not configured, the sweep runs locally with no change to behavior.

Reading the sweep detail page on Stage

The sweep detail page at /org/project/sweeps/<id> answers five questions a researcher needs to make decisions after a sweep: did it complete, which configurations won, where is the variance, what did it cost, and what should I look at next. The sections below describe what each part of the page shows and how to interpret it.

KPI strip

Directly below the sweep header is a row of six cells: total runs, completed, failed, running, total cost, and mean outcome. Numbers use the monospace font. The completed, failed, and running cells include a percentage of the total as a sub-label, so you can see at a glance whether the sweep is half done or nearly done without counting. The failed count turns red if any runs failed, which makes a partial failure visible without reading the matrix.

The mean outcome cell shows the mean of the primary outcome score across completed runs, plus a standard error (mean ± se). A high standard error relative to the mean means the signal is noisy and you probably need more trials per cell before drawing conclusions. A low standard error means the result is stable.

While the sweep is in progress (status "running"), the KPI strip polls every two seconds. The page stops polling when the sweep reaches a terminal state.

Matrix view

The matrix is the core visualization. It is a table whose rows and columns correspond to two sweep axes, defaulting to the two axes with the most distinct values. Use the "Rows" and "Cols" dropdowns above the matrix to switch which axes appear in each dimension. Any axes not shown in the matrix appear in the per-axis breakdown section instead.

Each cell in the matrix shows up to three pieces of information:

  • A row of small colored squares, one per run, showing each run's status: green for completed, red for failed, amber for running (pulsing), and gray for not yet started. If a cell has more than six runs, the first six are shown and a "+N" indicator counts the rest.
  • The mean primary outcome score in monospace type. When the cell has no completed runs, a dash appears instead.
  • A tiny horizontal box plot glyph when the "Show variance" toggle is on and the cell has two or more runs. The box marks the interquartile range, the center line marks the median, and the caps mark the minimum and maximum. A wide box means the result varies significantly across trials of the same configuration. A narrow box means the result is stable.

Cell background color shifts from a green tint for cells where all runs completed successfully to a red tint for cells where all runs failed. Mixed cells fall between those extremes. The tint is subtle enough not to interfere with the score text.

Hovering a cell shows a tooltip with the per-run breakdown: run ID prefix, status, score, and cost for each run. Clicking a single-run cell opens that run's detail page. Clicking a multi-run cell scrolls the page to the flat runs list and filters it to those runs.

When the sweep has more than 200 cells the matrix may be wide. It scrolls horizontally; the row and column headers stick to the left and top edges as you scroll.

Cost over time chart

Below the matrix is a line chart of cumulative USD spend over wall-clock time. The x-axis runs from the first run's start time to the last run's end time (or to "now" while the sweep is in progress). Each point on the line represents one run completing; the dot is green for a successful run and red for a failed one. The slope of the line tells you your burn rate. A sudden flat segment means runs completed cheaply or for free; a steep jump means a single expensive run landed.

If no runs recorded cost data, the section shows a brief message to that effect. Cost data is recorded when a run completes with a total_cost.usd field in its status update.

Per-axis breakdown

To the right of the cost chart is a compact table for each axis that is not currently assigned to the matrix rows or columns. Each table lists the axis's distinct values with the number of runs for that value, the mean outcome score, and the total cost. Clicking a row in any of these tables filters the flat runs list below to only runs matching that axis value.

The per-axis breakdown is where you discover that one axis does not matter. If "backend" shows identical mean scores across all values, the choice of backend is not a significant factor for this scenario. If "persona" shows a wide range, the scenario is sensitive to persona choice and you should investigate the outlier personas in the runs list.

When all axes are assigned to the matrix rows and columns, this section reads "All axes are shown in the matrix above."

Flat runs list

At the bottom of the page is a table of all runs in the sweep: ID, scenario, world, model, status, outcome score, cost, and duration. Each ID is a link to the run's detail page.

Clicking a matrix cell or a per-axis breakdown row applies a filter to this list and shows a banner above the table identifying the active filter. Clicking "Clear filter" returns to the unfiltered view. This is how you move from "the anthropic/enterprise_audit cell looks weak" to "here are the specific runs in that cell."

Actions

The action row above the KPI strip has three buttons. "Share" copies the current URL to the clipboard; the URL is already a permanent link. "Compare two runs" opens a picker where you select two runs from this sweep by their ID and opens the Stage comparison view. "Cancel sweep" appears only when the sweep status is "running"; clicking it asks for confirmation before posting the cancel request.
