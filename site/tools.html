Tools

A tool is a Python function the agent can call. This guide walks through the four shapes the @tool decorator takes, the return-envelope choices, and the optional capabilities a tool can opt into. Reach for it when you are adding a new tool or deciding which decorator shape fits the function you already have.

The bare decorator

Most tools want the bare form. Write a plain Python function with type hints and a docstring, stick @tool on top, and hand it to register_world. The decorator infers the tool's name from the function's __name__, the description from the first paragraph of the docstring, and the JSON-Schema parameters from the type hints. There is no configuration to repeat. 

from ensemble import register_world, tool


@tool
def lookup_user(user_id: str) -> dict:
    """Return the user record by id."""
    return {"id": user_id, "name": "Alice", "plan": "team"}


@tool
def search_kb(query: str, tags: list[str] | None = None) -> list:
    """Search the knowledge base. Tags filter the returned items."""
    return kb.search(query, tags or [])


register_world("my_world", tools=[lookup_user, search_kb])

Inference handles the parameter shapes you reach for most often. A str becomes {"type": "string"}, an int becomes {"type": "integer"}, a list[str] becomes an array of strings, and Optional[T] (or any parameter with a default of None) drops out of the required list while keeping its type. The inferred schema for search_kb above looks like this: 

{
  "type": "object",
  "properties": {
    "query": {"type": "string"},
    "tags": {"type": "array", "items": {"type": "string"}}
  },
  "required": ["query"]
}

Two rules cover the rest. Parameters without a default land in required; parameters with a default do not. Literal["a", "b"] becomes {"enum": ["a", "b"]}. Anything the inferrer cannot recognise produces an empty schema fragment, which the runtime treats as "any JSON value". When you want a tighter contract than that, reach for one of the override forms below.

The kwarg-override form

When inference is mostly right but one or two fields need a custom value, call the decorator with keyword arguments and leave the rest to inference. The common reasons are a tool name that differs from the function name, a description that is not the docstring's first paragraph, or a hand-rolled schema for a parameter that inference cannot reach. 

@tool(description="Submit the kernel for static checks and run.")
def submit(src: str):
    return runner.check_and_run(src)

Any of name, description, parameters, timeout_ms, resources, sandbox, and sandbox_world can be passed this way. Anything you omit is still inferred from the function. The kwarg form is the right call when you want one override without giving up the convenience of inference for everything else.

The four-argument factory

When the schema is already in hand, for example because it was generated from a shared spec or pulled off a remote server's tool list, call tool as a four-argument factory. This form does no inference; every field is explicit. It is also the right call for functions without type hints, where there is nothing for the inferrer to read. 

register_world("my_world", tools=[
    tool(
        "submit_kernel",
        "Submit the kernel and run it.",
        {"type": "object",
         "properties": {"src": {"type": "string"}},
         "required": ["src"]},
        submit_kernel_fn,
    ),
])

The arguments are name, description, the JSON-Schema parameters object, and the callable. The result is a PluginTool ready to drop into the tools= list. Reach for this form sparingly; the bare decorator is shorter for the cases where it works.

Subclass methods

Decorated methods on a World subclass behave the same way. The subclass walker picks them up at construction time and binds self automatically. The bare form works because the walker skips self when it infers the schema, so the parameters the agent sees are the real arguments only. 

from ensemble import World, tool


class MyWorld(World):
    def setup(self):
        self.db = Db()

    @tool
    def lookup_user(self, user_id: str) -> dict:
        """Return the user record by id."""
        return {"ok": True, "data": self.db.lookup_user(user_id)}

Use the subclass form when the tools need shared mutable state that is awkward to express with module-level closures. See Writing a world for the trade-offs between the subclass form and the flat register_world form.

Return shapes

A tool can return one of three shapes, and the wrapper figures out which one it got. The simplest is a plain dict, which the runtime sends back as the tool's effect. The next is an (effect, diff) tuple, where the diff is emitted as a state_diff event alongside the tool_result; reach for this when the tool mutates world state and you want the trace to record the delta. 

@tool
def issue_refund(user_id: str, amount_cents: int):
    """Issue a refund for the named user."""
    record = {"refund_id": f"r-{user_id}", "amount_cents": amount_cents}
    diff = {"refunds": {"+": [record]}}
    return record, diff

The third shape is the structured envelope, a dict that contains one or more of effect, diff, costs, and progress. The structured envelope is the right choice when the tool wants to annotate cost or emit progress without using the emit_progress injection path described below. 

@tool
def heavy_op(target: str):
    result = run(target)
    return {
        "effect": {"ok": True, "result": result},
        "costs": {"gpu_seconds": 4.2, "usd": 0.03},
    }

Argument routing tries fn(**args) first and falls back to fn(args) if the function rejects keyword expansion. The fallback is useful when the schema names a parameter that is not a legal Python identifier.

Optional capabilities

Five capabilities are available when a tool needs more than the bare contract. Each one is opt-in; tools that do not declare them pay no cost. The Tools reference covers the full contracts.

Timeouts

Pass timeout_ms when a single dispatch should cap at a known duration. The runtime emits a tool_timeout event when the deadline fires, the calling agent sees a tool error, and the scenario continues without waiting for the closure to finish. Pair timeouts with progress emission so the trace stays observable during the wait. 

@tool(timeout_ms=2000)
def slow(query: str) -> dict:
    """Search a remote index that occasionally hangs."""
    return remote.search(query)

Resources

Pass resources=[...] when a tool needs to serialise with other dispatches against a shared lock, like a billing database or a GPU lane. Each name is a permit the runtime acquires before the closure runs and releases when it returns. Two dispatches that share a resource name serialise through the world's ResourceManager; unrelated names run in parallel. Declare permits up front in the world manifest to allow more than one concurrent holder. 

@tool(resources=["billing_db"])
def lookup(user_id: str) -> dict:
    """Read the billing record. Holds the billing_db permit."""
    return billing.lookup(user_id)

Progress

Declare an emit_progress parameter to receive a callable that records progress events. Each invocation logs one (fraction, message) pair, which the runtime flushes to the trace as a progress event right before the trailing tool_result. The helper inspects the function's signature, so tools that do not need progress reporting simply omit the parameter. 

@tool
def reconcile_billing(user_id: str, months: int, emit_progress) -> dict:
    """Walk a user's billing history month by month."""
    for i in range(1, months + 1):
        emit_progress(i / months, f"scanned month {i}/{months}")
        reconcile_one(user_id, i)
    return {"ok": True, "months_reconciled": months}

Costs

Annotate cost by returning a costs dict in the structured envelope. Units are open strings, each entry becomes one cost event on the trace attributed to the calling actor, and the runtime maintains running totals per unit. A budget declared via world.set_budget(unit, amount, actor=...) halts the scheduler when a recorded cost would push the total past the cap. 

@tool
def render_video(frames: int) -> dict:
    """Render a clip. Reports gpu seconds against the calling actor."""
    duration = render(frames)
    return {
        "effect": {"ok": True, "frames": frames},
        "costs": {"gpu_seconds": duration},
    }

Sandbox

Pass sandbox=True when a tool runs untrusted code, leaves CUDA contexts in a bad state, or imports a binary that occasionally segfaults. Each dispatch runs in a fresh python -m ensemble.tool_worker subprocess that reimports the world's package, calls the tool with the JSON args, prints the result, and exits. A crash in the worker surfaces as a structured error envelope rather than killing the scheduler. The cost is process startup per call, so sandbox mode pays off only when isolation is worth more than the latency. 

@tool(sandbox=True)
def compile_kernel(src: str) -> dict:
    """Compile and load a CUDA kernel. Sandboxed to survive crashes."""
    return kernel.compile(src)

A sandboxed tool must encode its entire input in its JSON args and its entire output in its JSON return. State the parent's closures held is not shared with the worker; for configuration the worker needs to read, use world.shared_state, which the runtime serialises into ENSEMBLE_SHARED_STATE on every dispatch.
