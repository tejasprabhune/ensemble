state

An in-memory SQLite database with five tables:

users         (id, name, email, plan)
tickets       (id, user_id, subject, status, opened_at)
subscriptions (user_id, plan)
kb            (id, title, body)
audit_log     (id, ts, who, action, payload)
refunds       (id, user_id, amount_cents, reason, ts)

State is bundled into a WorldState trait impl in Rust; the scenario authors only see the tools that operate on it.

tools

  • lookup_user(user_id) — fetch a user record
  • lookup_ticket(ticket_id) — fetch a ticket
  • issue_refund(user_id, amount_cents, reason) — one per user per run; second call errors
  • escalate(ticket_id, to_team) — sets status to escalated
  • search_kb(query) — case-insensitive substring search
  • update_subscription(user_id, plan) — switch a user's plan

scenarios shipped

  • plank.single_ticket — 1 user, 1 agent. Sanity check.
  • plank.refund_storm — 3 users, 2 agents. The hero demo on the front page.
  • plank.enterprise_audit — async with path with mid-run intervention.

run it

cargo build --release
ensemble run plank.refund_storm --world plank
ensemble trace view traces/plank_refund_storm.jsonl
