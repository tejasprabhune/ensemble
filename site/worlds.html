Writing a world

A world is the typed, mutable environment your scenarios run inside. It owns the tools the agents call, the predicates the graders ask, and any state that needs to persist across turns. Most worlds are a single Python file; a small subset needs a Rust core for typed snapshot and restore.

The pure-Python path

Reach for this first. The scaffold lays down a single my_world.py with an example tool, a world.toml manifest, and a runnable smoke scenario. You add tools by decorating plain Python functions; the framework infers the JSON schema from the type hints. 

ensemble init my_world
cd my_world
ensemble run my_world.smoke

The scaffolded module looks like this: 

# my_world.py
from ensemble import register_world, tool


@tool
def echo(text: str) -> str:
    """Return the text back. Replace me with a real tool."""
    return text


register_world(
    "my_world",
    tools=[echo],
    default_agent_model="claude-sonnet-4-5",
)

A real world replaces echo with the tools your scenarios need. Closure-captured state survives across dispatches in the same process: 

from ensemble import register_world, tool

_db = {
    "u-alice": {"name": "Alice Chen", "plan": "team"},
    "u-bob":   {"name": "Bob Wong",   "plan": "starter"},
}
_tickets = {}


@tool
def lookup_user(user_id: str) -> dict:
    """Return the user record by id."""
    return _db.get(user_id, {"error": "unknown user"})


@tool
def open_ticket(user_id: str, subject: str) -> dict:
    """Open a support ticket for a user."""
    ticket_id = f"t-{len(_tickets) + 100}"
    _tickets[ticket_id] = {"user_id": user_id, "subject": subject, "status": "open"}
    return {"ticket_id": ticket_id, "status": "open"}


register_world("my_world", tools=[lookup_user, open_ticket])

For stateful worlds, the factory form above scales fine: capture the state in closures or module-level dicts as shown. The framework also supports a class MyWorld(World) subclass form for scenarios that construct the world themselves rather than going through @scenario(..., world="name"); the World API reference covers it. The factory form is the recommended path for scenarios using the standard decorator.

The world.toml manifest

Every world ships a world.toml at the root of its directory. The CLI reads it to discover the world's Python package, default models, default personas, and any shared resources. The scaffold produces a minimal one; here is the full schema: 

[world]
name = "my_world"
python_package = "my_world"
default_user_model = "claude-haiku-4-5"
default_agent_model = "claude-sonnet-4-5"
personas_dir = "personas"     # optional, relative to manifest
rust_crate = "world"          # optional, only when --with-rust

[[world.default_personas]]
name = "frustrated_customer"

[world.resources]
billing_db = { permits = 1 }
inference_pool = { permits = 4 }

The default-model fields let you set per-world preferred models so scenarios do not have to repeat the same model= string on every spawn. The resources table declares shared locks the runtime acquires before dispatching a tool that names them.

When you run a scenario from inside the world's directory, the CLI auto-discovers the manifest and registers the world. No ensemble worlds add step is required for the common flow. The World API reference covers the full manifest schema and the registry semantics for worlds that live outside the cwd.

Predicates

Predicates are the yes/no questions the world exposes for scenarios and graders to ask. A predicate walks the trace and returns a bool. Use them when the answer is "did the scenario end in the state I consider good". 

from ensemble import predicate, register_world, tool


@tool
def issue_refund(user_id: str, amount_cents: int) -> dict:
    """Issue a refund for the named user."""
    return {"refund_id": f"r-{user_id}", "amount_cents": amount_cents}


@predicate(name="had_double_refund")
def had_double_refund(trace, args):
    """True iff issue_refund was called twice for the same user."""
    seen = set()
    for ev in trace:
        if ev["payload"].get("name") == "issue_refund":
            user_id = ev["payload"]["args"]["user_id"]
            if user_id in seen:
                return True
            seen.add(user_id)
    return False


register_world(
    "my_world",
    tools=[issue_refund],
    predicates=[had_double_refund],
)

Scenarios reference predicates by name in graders and in until_predicate stop conditions. See the Predicates reference for the contract, including how per-user predicates dispatch on args["user_id"].

When to reach for Rust

The pure-Python path covers the dominant case. The heavyweight path adds a Rust crate that implements WorldState, and is the right choice when:

  • The world's state needs strongly-typed snapshot and restore semantics. Agora uses SQLite with typed tables; checkpointing the simulation in Rust is straightforward. Doing the same in pure Python means pickle, which is fragile for cross-version replay.
  • The world's state is large enough that copying it across the Python-Rust boundary on every tool call matters. The rust-state path lets the runtime keep the canonical state in Rust and hand back diffs.
  • You want the typed tool-call enum that WorldState ::ToolCall gives you, with compile-time exhaustiveness when the runtime adds a new tool kind.

Scaffold with ensemble init my_world --with-rust. The result is the same Python plugin plus a Cargo crate at world/ with a WorldState impl ready to fill in. The World API reference walks through the trait and the Python glue.

Shared state and sandboxed tools

world.shared_state is a mutable JSON-serialisable dict that the runtime forwards to sandbox-mode tool workers. Reach for it when a tool runs in a subprocess (because it imports torch, or because it might segfault) and needs to read configuration the parent set. Mutations in the worker do not propagate back; the dict is a one-way configuration channel.

Sandboxed tools run each dispatch in a fresh python -m ensemble.tool_worker subprocess. The worker reimports the world's package, looks up the tool, and calls it once with the supplied JSON args. Use sandboxing for tools that can crash the interpreter; the cost is process startup per call. See the Tools reference for the full contract.
