Writing a scenario

A scenario is the script that arranges a single run inside a world. It spawns the users and agents, hands the agent its first problem, declares when the run should stop, and reports what counts as success. Most scenarios are twenty lines of Python around a single @scenario decorator.

The minimal shape

Every scenario follows the same skeleton: import the world so its plugin registers, decorate an async function with @scenario, spawn the actors, yield an Until condition that tells the scheduler when to stop, and yield a grader dict with the scores. The function takes one argument, the world instance the runner constructs for you. 

import my_world  # noqa: F401  registers the world plugin
from ensemble import scenario


@scenario("my_world.refund", world="my_world")
async def refund(world):
    alice = world.spawn_user(persona="frustrated_customer")
    rep = world.spawn_agent(tools=["lookup_user", "issue_refund"])

    alice.say(rep.id, "I want a refund for the last three months.")

    yield world.until(world.turn_count > 20)
    yield {"refund_issued": 1.0 if alice.hidden_goal_resolved() else 0.0}

The decorator's first argument is the registry name, by convention <world>.<short_name>. The world= keyword names the world the runner builds when --world is not passed on the CLI. Both ensemble run my_world.refund and run_scenario("my_world.refund") look the scenario up by that registry name.

Importing the world's Python package at the top of the file is what triggers its register_world call. The World the runner constructs cannot find the plugin otherwise, so this import is load-bearing even when you do not reference it directly.

Spawning users and agents

world.spawn_user creates a user actor backed by a persona. world.spawn_agent creates an agent backed by the world's LLM backend and tool registry. Both return a proxy you hold onto so the scenario can speak as the actor and read its state later. 

alice = world.spawn_user(persona="frustrated_customer", hidden_goal="refund_3mo")
rep = world.spawn_agent(tools=["lookup_user", "issue_refund"])

alice.say(rep.id, "I want a refund for the last three months.")

You can leave the id off both calls. The runtime assigns user-1, user-2, ... and agent-1, agent-2, ... from a per-world counter, so scaffold scenarios stay short and only name actors when graders or stop conditions need to reference them by id.

The model argument is also optional. When you omit it, spawn_user uses the world's default_user_model and spawn_agent uses its default_agent_model, both set in register_world or in world.toml. Override on a single actor by passing model="claude-sonnet-4-5" directly.

Some scenarios have no real user: the agent receives an opening problem statement and iterates on its own until it emits a done signal. For that shape, reach for world.opener instead of spawning a user. The opener sends a seed message without creating a user actor, so it never appears in world.users and the trace records a kickoff event rather than a user_spawned one. 

@scenario("my_world.refactor", world="my_world")
async def refactor(world):
    rep = world.spawn_agent(tools=["read", "edit", "grep", "run_tests"])
    world.opener("rename foo to bar in this file and fix call sites", to=rep.id)

    yield world.until_done(rep.id)
    yield {"completed": 1.0}

Until conditions

The first thing a scenario yields is an Until: the question the scheduler asks every tick to decide whether to keep running. The four building blocks cover almost every scenario.

world.turn_count > N is the simplest. The turn_count attribute is a sentinel that produces an Until when you compare it with > or >=. Use this when the scenario should run for a fixed budget regardless of what happens inside. 

yield world.until(world.turn_count > 20)

world.until_predicate("name", **args) stops the run when a named world predicate becomes true on the live trace. Reach for this when the natural answer to "are we done" is a fact about what the agent did, not a turn budget. The world has to register the predicate; passing an unknown name raises immediately, so typos surface at scenario definition rather than as a run that never stops. 

yield world.until_predicate("submit_called")

world.until_agent_emits halts when a specific agent says something that matches a criterion. Exactly one of contains, equals, or regex is required. world.until_done(actor_id) is the shorthand for the silent-loop case: it watches for the agent emitting "DONE", and you can override the sentinel with signal=. 

yield world.until_agent_emits(rep.id, contains="ready for review")
yield world.until_done(rep.id, signal="FINISHED")

Real scenarios usually want a stop condition with a safety net: stop on the thing you care about, but also cap the budget so a misbehaving agent does not run forever. Combine conditions with | (any-of) and & (all-of). The same combinators are available as any_of and all_of when you want function-call form for nested expressions. 

from ensemble import until_predicate

yield world.until(world.until_predicate("submit_called") | (world.turn_count > 30))
yield world.until(world.until_done(rep.id) | (world.turn_count > 100))
yield world.until((world.turn_count >= 4) & until_predicate("hit_goal"))

Two scenario flavors

Scenarios come in two shapes. The common case is an async generator that yields its until condition, lets the runner drive the scheduler, and then yields the grader dict. Pick this whenever the scenario commits to a single stop condition before the run starts. 

@scenario("agora.smoke", world="agora")
async def smoke(world):
    user = world.spawn_user(persona="patient_retail")
    rep = world.spawn_agent(tools=["search_kb"])
    user.say(rep.id, "hi")

    yield world.until(world.turn_count > 8)
    yield {"saw_event": 1.0}

The other shape is a regular async function (no yield) that drives the run itself through async with world.simulate(). The context manager starts the scheduler on a background task and gives you a SimulationRun handle with await run.wait_until(condition, timeout_ms=...). Inside the block, the scenario can inspect state, push new messages from a user, or mutate the world between waits. Pick this when the scenario needs to react to what already happened. 

@scenario("agora.audit", world="agora")
async def audit(world):
    carol = world.spawn_user(persona="enterprise_admin")
    rep = world.spawn_agent(tools=["escalate"])
    carol.say(rep.id, "audit log please")

    async with world.simulate() as run:
        await run.wait_until(world.turn_count > 10)
        carol.say(rep.id, "escalate this")
        await run.wait_until(world.turn_count > 18)

    return {"escalated": 1.0 if carol.hidden_goal_resolved() else 0.0}

The simulate flavor returns the grader dict from a plain return statement; there is no second yield. The yield flavor is shorter and reads top-to-bottom; reach for the simulate form only when the scenario has a real reason to inject behavior in the middle of the run.

Graders

The grader dict is how a scenario reports outcomes. The keys are score names; the values are floats, by convention 0.0 or 1.0 for boolean outcomes and any float for continuous ones. The same dict ends up in the run's scores field and in the runs index, so keep the names stable across runs you intend to compare. 

yield {
    "refund_issued": 1.0 if alice.hidden_goal_resolved() else 0.0,
    "no_double_refund": 1.0 - float(world.evaluate_predicate("had_double_refund")),
}

Most graders are built from world predicates and per-user helpers. world.evaluate_predicate("name") runs a registered predicate against the final trace and returns a bool. The user proxy exposes shortcuts like alice.hidden_goal_resolved() and alice.predicate("name") for per-user predicates that filter on args["user_id"]. Unknown predicate names raise rather than silently returning false, so a grader typo fails loudly.

The Predicates reference covers the full predicate contract, including the per-user dispatch rules and how the trace argument is shaped.

The declarative TOML form

For static scenarios that do not need mid-run intervention, you can write scenarios.toml instead of Python. ensemble.load_manifest(path) parses the file and registers one scenario per entry, so the same runner code drives both Python and TOML scenarios. 

# examples/agora/scenarios.toml
[scenario.refund_storm]
world = "agora"
duration_turns = 30
seed = 42

[[scenario.refund_storm.users]]
id = "alice"
persona = "frustrated_power_user"
hidden_goal = "refund_3mo"
initial_action = { tool = "open_ticket", args = { ticket_id = "t-100", user_id = "u-alice", subject = "want my money back" } }

[[scenario.refund_storm.agents]]
id = "rep1"
model = "claude-sonnet-4-5"
tools = ["lookup_user", "issue_refund", "escalate", "search_kb"]

[scenario.refund_storm.graders]
alice_refund_resolved = "alice_hidden_goal_resolved"
global_no_double_refunds = "not had_double_refund"

duration_turns becomes the world.turn_count > N stop condition. users and agents are arrays that forward to spawn_user and spawn_agent. The optional initial_action on a user fires a tool call with no actor attribution at startup, useful for seeding world state before the conversation begins.

Grader expressions live in a small boolean DSL: bare predicate names, and, or, not, parens, and the literals true and false. Per-user predicates are addressable as <user_id>_<predicate>, evaluated with args = {"user_id": "<user_id>"}. Truthy expressions become 1.0, falsy 0.0. Reach for TOML when the scenario is a fixed configuration you want to enumerate across a sweep; reach for Python when the scenario needs to do anything the DSL cannot express.
